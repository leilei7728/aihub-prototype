# CC Switch 统一代理层需求文档

**日期：** 2026-07-10  
**需求方向：** 简化供应商配置，实现一次配置、多工具通用

---

## 一、需求背景与目的

当前 CC Switch 对每个 AI 工具（Claude Desktop / Codex / Gemini / OpenCode / OpenClaw / Hermes / Claude Code）需要单独配置供应商，不同工具的配置界面字段各不相同，用户体验割裂、重复填写成本高。

**目标：用户只需配置一次 API Key，所有工具自动可用。**

---

## 二、核心交互流程

```
1. 用户填写：供应商名称 + API Key + 请求地址（官网链接可选）
        ↓
2. 系统自动请求 /v1/models，拉取该 Key 下所有可用模型
        ↓
3. 本地启动统一代理服务，同时暴露两套协议端点：
        ├── 端口 A：Anthropic Messages 协议
        └── 端口 B：OpenAI Chat Completions 协议
        ↓
4. CC Switch 自动将对应代理地址写入各工具配置：
        ├── Claude Desktop / Claude Code → 指向端口 A（Anthropic 协议）
        └── Codex / Gemini / OpenCode / OpenClaw / Hermes → 指向端口 B（OpenAI 协议）
        ↓
5. 代理层完成协议格式转换后，将请求统一转发到用户配置的真实 API 端点
```

---

## 三、用户配置界面（极简化）

| 字段 | 是否必填 | 说明 |
|---|---|---|
| 供应商名称 | 必填 | 用于界面展示，用户自定义 |
| API Key | 必填 | 鉴权凭证，支持聚合型 Key（含多家厂商模型） |
| 请求地址 | 必填 | 供应商 API 端点，如 `https://api.deepseek.com` |
| 官网链接 | 可选 | 仅用于展示，不影响功能 |

配置完成后，系统自动拉取模型列表，用户确认后即完成全部配置。

### 模型启用与槽位映射的分层设计

**第一层：Key 配置阶段（无槽位限制）**
- 系统拉取该 Key 下所有模型，用户勾选要启用的模型
- 所有启用的模型统一映射到代理端点，数量不限
- Codex / Gemini / OpenCode 等工具直接使用，无需额外配置

**第二层：接入 Claude Desktop 时（按需配置）**
- Claude Desktop 仅支持 Sonnet / Opus / Haiku 3 个角色槽
- 用户在"AI 工具"页面选择接入 Claude Desktop 时，才出现槽位绑定步骤
- 从已启用的模型中各选一个绑定到对应槽位即可
- 槽位概念不在 Key 配置阶段暴露，不影响其他工具的使用体验

## 四、技术实现要点

### 4.1 模型列表拉取

- 调用 `GET /v1/models` 接口，自动获取该 Key 下所有可用模型
- 主流聚合 API（OpenRouter、DeepSeek、硅基流动等）均支持此接口
- **兜底方案：** 不支持 `/v1/models` 的供应商（如 Anthropic 原生），由技术人员维护一份线上 JSON 模型预设列表，CC Switch 启动时自动拉取，用户无需任何操作

### 4.2 协议双向转换

代理层需实现 Anthropic ↔ OpenAI 格式互转，重点处理以下差异：

| 差异点 | Anthropic 格式 | OpenAI 格式 |
|---|---|---|
| system 字段 | 独立顶层字段 | 放入 messages 数组首条 |
| 工具调用 | `tool_use` / `tool_result` | `function_call` / `tool_calls` |
| 流式响应 | `content_block_delta` | `choices[0].delta` |
| 模型名格式 | `claude-sonnet-*` 角色 ID | 直接模型名 |

### 4.3 本地代理服务

- 随 CC Switch 启动/关闭，进程生命周期与主程序绑定
- 本地监听两个独立端口：
  - 端口 A：Anthropic Messages 协议
  - 端口 B：OpenAI Chat Completions 协议
- 所有请求在代理层完成格式转换后，统一转发到用户配置的真实 API 端点

### 4.4 各工具配置自动写入

CC Switch 在用户完成供应商配置后，自动将本地代理地址写入各工具配置文件：

| 工具 | 配置写入方式 |
|---|---|
| Claude Desktop | 系统代理拦截 + 模型映射 |
| Claude Code | 修改请求地址为本地代理 |
| Codex | 写入 `config.toml` 的 `base_url` |
| Gemini | 写入 `.env` 的 `GEMINI_BASE_URL` |
| OpenCode / Hermes / OpenClaw | 写入各自 JSON 配置的 Base URL 字段 |

用户无需感知各工具配置文件的位置和格式。

## 五、参考开源方案

协议转换层**不需要从零开发**，可复用以下成熟项目的实现：

- **LiteLLM** — 支持 100+ 供应商的统一代理，Anthropic/OpenAI 互转成熟
- **one-api / new-api** — 国内广泛使用的 API 聚合网关，格式转换完整

建议直接集成或参考其转换逻辑，减少开发工作量。

## 七、边界场景处理方案

### 7.1 本地代理端口冲突处理

**方案：** 固定默认端口 + 冲突时自动顺延到下一个可用端口。

- 预设默认端口（如 Anthropic 协议用 `11430`，OpenAI 协议用 `11431`）
- 启动时检测端口是否被占用，若冲突则自动顺延（+1）直到找到可用端口
- 端口确定后自动同步写入各工具配置，用户无感知
- 在 CC Switch 设置页展示当前实际占用端口，供高级用户参考

### 7.2 Anthropic 原生 Key 的模型列表维护

**方案：** 由技术人员维护一份线上 JSON 配置文件，Anthropic 发布新模型时需手动更新，CC Switch 启动时自动拉取同步，用户无感知。

### 7.3 多供应商并存时的路由策略

**方案：** 按模型名路由，每个模型绑定到对应供应商。

- 用户拉取模型列表时，每个模型记录来源供应商
- 代理层收到请求后，根据请求中携带的模型名，查找对应供应商的 Key 和端点
- 不同供应商存在同名模型时（如都有 `deepseek-v3`），模型列表以 `供应商/模型名` 格式展示（如 `DeepSeek / deepseek-v3`、`OpenRouter / deepseek-v3`），用户选哪个即路由到对应供应商，系统自动处理，无需用户配置优先级
- 路由表随供应商增删自动更新
