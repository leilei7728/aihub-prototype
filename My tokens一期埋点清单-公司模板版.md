# My tokens 一期埋点清单

版本：v1.0.0  
日期：2026-07-15  
用途：将 [My tokens一期埋点方案.md](产品原型/CC本地部署/My%20tokens%E4%B8%80%E6%9C%9F%E5%9F%8B%E7%82%B9%E6%96%B9%E6%A1%88.md) 中的核心事件，转换为公司常用的埋点台账格式，便于复制到表格中继续补充截图、状态、版本号。

说明：

1. 以下内容按表头顺序解释每个字段的含义、作用和填写方式。
2. 目的不是讲埋点原理，而是让看表的人知道“这一列是什么、为什么要填、应该怎么填”。

## 字段说明

### 1. 序号

1. 含义：表格内的行号。
2. 作用：方便沟通、核对和定位具体事件。
3. 取值：按 1、2、3 递增即可，不参与埋点上报。

### 2. 所属模块

1. 含义：事件所属的页面、弹层或业务模块。
2. 作用：帮助快速判断这条埋点属于哪一块功能。
3. 取值：使用清晰的中文模块名，如“登录弹层”“AI 工具管理”“订阅 / 充值”。

### 3. 事件名称

1. 含义：给人看的事件名称。
2. 作用：让产品、测试、研发、数据都能直接看懂这条埋点在记录什么。
3. 命名：建议使用“业务场景-触发对象-动作结果”，如“登录弹层-登录成功”“AI工具管理-工具安装失败”。

### 4. 事件Key

1. 含义：给系统识别的事件标识。
2. 作用：用于前后端上报、平台配置和数据入库。
3. 命名：本表使用英文小写加下划线；如团队已有规范，可按团队规范执行，但同一份表内要保持一致。

### 5. 事件/场景ID

1. 含义：当前事件或场景的唯一编号。
2. 作用：方便台账管理、链路引用和后续扩展。
3. 命名：本表建议使用 `MT-模块缩写-三位流水号`，如 `MT-AUTH-001`、`MT-TOOL-004`；新增事件在原模块内继续递增。

### 6. 上级ID

1. 含义：当前事件关联的上级编号。
2. 作用：表示当前事件挂靠在哪个上游节点上，方便看完整链路。
3. 取值：一级入口或页面曝光填 `0`；后续动作、成功、失败、异常等事件填写触发它的上级编号，可对应事件或场景。

### 7. 页面/位置

1. 含义：事件发生的页面区域、弹层区域或控件位置。
2. 作用：帮助产品、测试、研发快速定位埋点所在位置。
3. 取值：写清“页面 / 弹层 + 区域”，如“Tokens 管理页顶部余额区”“安装失败引导弹层”。

### 8. 发送时机

1. 含义：这条埋点在什么时刻上报。
2. 作用：避免同一事件在不同时间点被重复理解或错误接入。
3. 取值：只写“什么时候发”，如“页面完成渲染后上报”“点击并触发动作后上报”“确认成功后上报”，不在这一列写统计规则和业务目的。

### 9. 统计类型

1. 含义：这条埋点属于哪种统计类别。
2. 作用：帮助区分曝光、操作、结果、异常这几类事件。
3. 取值：本表统一使用 `页面浏览上报-单行为`、`操作上报-单行为`、`结果上报-单行为`、`异常上报-单行为` 四类。

### 10. 数据统计类型

1. 含义：这条埋点上报的是单纯次数，还是次数加数值。
2. 作用：帮助确认后续统计口径和字段设计。
3. 取值：只记次数填 `计数`；如果还要带金额、次数、时长等数值，填 `计数+数值`。

### 11. 上报参数

1. 含义：事件上报时需要一起带上的字段。
2. 作用：补充事件上下文，支持后续筛选、分析和排查。
3. 取值：先写核心参数名即可，如 `tool_name`、`order_id`、`fail_reason`；所有核心事件默认带公共参数。

#### 公共参数说明

1. 含义：公共参数指所有核心事件默认统一上报的一组通用字段。
2. 作用：用于补充用户、设备、版本、来源等基础上下文，方便后续做事件关联、版本排查、来源分析和问题定位。
3. 建议字段：可优先包含 `user_id`、`device_id`、`client_version`、`os_type`、`event_time`、`channel_source`、`page_source`。
4. 使用方式：主表中写“+ 公共参数”时，表示该事件除当前列出的业务参数外，还需默认带上这组通用字段。
5. 说明：如团队已有统一埋点公共字段规范，以团队现有规范为准；如无统一规范，建议至少先约定一版基础字段清单。

### 12. 统计口径

1. 含义：这条数据按什么规则统计。
2. 作用：避免不同角色对同一指标理解不一致。
3. 取值：只写“怎么算”，如“每点击一次上报一次”“以后端口径为准”“以客户端口径为准”，不在这一列写触发时点。

### 13. 优先级

1. 含义：这条埋点在当前版本中的接入优先顺序。
2. 作用：帮助研发排期、联调和验收时先抓核心链路，再补支撑项和兜底项。
3. 取值：本表建议使用 `P0`、`P1`、`P2`；其中 `P0` 为一期必须优先接入，`P1` 为重要支撑项，`P2` 为补充观察项或兜底项。

### 14. 接入状态

1. 含义：这条埋点当前的接入进度。
2. 作用：用于跟踪研发、测试、验收推进情况。
3. 取值：可按团队习惯填写；本表默认使用 `待接入`，后续也可扩展为 `接入中`、`已接入`、`已验收`。

### 15. 所属版本

1. 含义：这条埋点归属的上线版本。
2. 作用：方便区分不同阶段新增或调整的埋点。
3. 取值：统一使用版本号，如 `v1.0.0`、`v1.0.1`；本表当前统一为 `v1.0.0`。

### 16. 补充说明

1. 含义：对事件的补充说明。
2. 作用：记录表头其他字段没有表达、但接入和验收时又容易遗漏的信息。
3. 取值：优先写前置条件、分支说明、特殊限制、异常兜底、依赖关系；不重复写事件名、发送时机、口径里已经明确的内容。

### 17. 验收标准

1. 含义：如何判断这条埋点已经接对了。
2. 作用：给测试、产品、研发一个统一的验收标准。
3. 取值：只写最终要验证的结果，如“能统计登录成功率”“能区分下载成功和安装成功”“能定位支付成功但未生效问题”，不写接入限制和实现细节。

## 使用原则

### 1. 这张表不是只看转化

本表按“转化 + 诊断”两层来设计：

1. 转化事件：回答用户有没有走到下一步，如曝光、点击、提交、成功。
2. 诊断事件：回答用户为什么没走通，如失败、异常、未生效、恢复动作。

### 2. 主链路至少要成对出现

一期核心链路，不建议只写成功，不写失败。关键节点至少要能同时回答“有没有成功”和“失败时卡在哪里”。

1. 登录：登录成功 + 登录失败。
2. 工具安装：安装成功 + 安装失败 + 失败后引导。
3. 工具启动：启动成功 + 启动失败 + 失败后引导。
4. 支付：订阅 / 充值支付成功 + 支付失败。
5. 权益生效：生效成功 + 生效失败。
6. API Key：配置成功 + 配置失败 + 首次调用成功 / 失败。

### 3. 什么时候只记转化，什么时候要补诊断

1. 只看页面到达、按钮点击、提交量时，优先看转化事件。
2. 只要某一步存在失败、拦截、未生效、恢复动作，就需要补诊断事件。
3. 如果一个结果失败后会触发新的引导弹层或补救动作，这个引导或补救动作也应记录。

### 4. 诊断不等于把所有失败拆成很多事件

一期默认优先使用“统一失败事件 + 原因参数”的做法，不建议把每一种失败都拆成独立事件。

1. 先保留一个失败事件，如 `auth_login_fail`。
2. 再通过 `fail_reason`、`fail_message`、`exception_scene` 这类参数区分失败类型。
3. 只有当某类失败需要单独分析、单独验收、单独触发补救时，再考虑拆成独立事件。

### 5. 参数分工怎么理解

1. 产品侧负责写清核心业务参数，也就是后续分析必须依赖的字段。
2. 研发侧负责补齐公共技术参数，如时间戳、设备信息、客户端版本、系统版本、trace_id 等。
3. 如果主表未写清核心业务参数，不建议默认由研发自行补齐。

## 参数取值说明

以下内容不是要求每条事件都填写，而是用来说明：当事件涉及失败、异常、恢复时，相关参数一般怎么取值、没覆盖时怎么兜底。

### 1. fail_reason

1. 作用：用于区分失败的主要原因。
2. 适用：登录失败、安装失败、启动失败、支付失败、API Key 校验失败、调用失败等。
3. 写法：建议使用稳定、可枚举的短值，不要直接使用整段报错文案。
4. 常见示例：`network_error`、`invalid_code`、`agreement_unchecked`、`account_not_exist`、`permission_denied`、`system_error`、`payment_cancelled`、`insufficient_balance`、`invalid_api_key`。
5. 兜底要求：如果实际失败原因未覆盖在既有枚举内，不要不上报，统一上报 `other`，同时补充 `fail_message`。
6. 登录失败推荐取值：`network_error`、`invalid_code`、`agreement_unchecked`、`password_error`、`account_not_exist`、`login_expired`、`system_error`、`other`。

### 2. fail_message

1. 作用：保留原始失败提示或返回文案，方便排查。
2. 适用：需要保留接口返回提示、客户端报错文案时。
3. 写法：允许保留原文，但不建议把它当成统计分类字段。

### 3. exception_scene

1. 作用：说明异常发生在哪个业务场景，而不是只说明为什么失败。
2. 适用：安装、启动、同步模型、API Key 调用、全局异常弹层等。
3. 常见示例：`install_process`、`launch_process`、`post_payment_first_call`、`api_key_validation`、`model_sync`。

### 4. resolve_method

1. 作用：说明异常后给用户提供了什么恢复方式。
2. 适用：失败引导弹层、异常恢复按钮、客服兜底入口等。
3. 常见示例：`retry`、`redownload`、`reinstall`、`go_to_payment`、`go_to_api_key_config`、`contact_support`。

### 5. 使用边界

1. `fail_reason` 解决“为什么失败”。
2. `exception_scene` 解决“失败发生在哪个阶段”。
3. `resolve_method` 解决“失败后给了什么恢复动作”。
4. 如果只需要知道失败总量，保留失败事件即可；如果需要定位原因，至少补 `fail_reason`。

## 失败原因推荐取值

以下内容用于帮助产品、研发、测试对齐失败类事件的常用写法。不是要求一次性列全所有情况，而是先给一期核心链路一套可执行的推荐值。

### 1. 登录失败

对应事件：`auth_login_fail`

## 登录失败怎么体现具体原因

登录失败的具体原因，不建议拆成很多独立事件，统一在 `auth_login_fail` 这一条里体现。

1. 主表体现位置：登录失败这一行的“上报参数”里明确写 `fail_reason`、`fail_message`。
2. 参数说明体现位置：在“参数取值说明”和“失败原因推荐取值”里补 `fail_reason` 的推荐取值。
3. 接入要求：研发至少要保证失败时上报 `fail_reason`；如果命中不了既有枚举，也必须上报 `other`，不能留空。
4. 排查兜底：当 `fail_reason = other` 时，需同时上报 `fail_message` 原始提示，避免数据不可用。
